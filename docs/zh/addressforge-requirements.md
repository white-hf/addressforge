# Address Library AddressForge Open Source Platform Requirements

## 1. 文档目的

这份文档定义的是一个**开源、自部署、可训练、可对外提供地址解析 API** 的地址系统需求。

它不是一个多租户 SaaS，也不是一个统一托管平台。它的形态是：

- 默认提供一套已经训练好的加拿大 / 北美地址模型
- 其他开发者可以下载代码，自行部署
- 其他开发者可以接入自己的地址数据
- 其他开发者可以训练自己的模型
- 其他开发者可以用自己的模型清洗自己的地址库，并提供自己的地址解析 API

这套系统要做到“麻雀虽小，五脏俱全”：

- 能离线清洗
- 能在线解析
- 能持续学习
- 能版本化管理
- 能人工审核
- 能对外提供标准 API

## 2. 产品定位

### 2.1 先做什么

系统先聚焦：

- 加拿大
- 北美
- 以 Halifax / Nova Scotia 为默认基础样例

原因很简单：

- 这套系统已经在加拿大地址上积累了完整数据链路
- 加拿大 / 北美地址规律比较典型，适合作为默认模型
- 先把一个区域做深做透，比一开始追求全球更现实

### 2.2 不做什么

当前版本不以“全球地址统一模型”为目标。

不优先做：

- 全球所有国家一次性覆盖
- 中央云托管多租户平台
- 复杂计费系统
- 统一账户体系

这套系统的目标是提供一个**标准、可复用、可修改** 的开源地址智能系统，让别人在自己的环境里做自己的国家 / 地区地址处理。

## 3. 核心目标

系统最终要同时满足四类目标：

1. **地址解析**
   - 把脏地址拆成结构化字段
   - 识别 unit、street、building、city、province、postal code 等信息

2. **地址清洗**
   - 标准化格式
   - 去噪
   - 校验缺失
   - 识别冲突

3. **地址生产**
   - 生成 canonical building / unit 地址库
   - 沉淀用户历史地址事实
   - 保留证据链和版本

4. **地址 API**
   - 对外提供解析、标准化、校验、建议补全、置信度等接口
   - 让其他开发者可以直接调用

## 4. 用户是谁

这套系统面向两类使用者：

### 4.1 系统使用者

他们会直接部署系统并使用：

- 内部数据工程师
- 业务工程师
- 数据平台工程师
- 需要地址解析能力的应用开发者

### 4.2 系统改造者

他们会下载代码后，按照自己的国家 / 地区数据做修改：

- 接入自己的原始地址数据
- 接入自己的标准参考库
- 训练自己的模型
- 提供自己的地址解析 API

## 5. 功能需求

### 5.1 默认加拿大模型

系统必须默认附带：

- 加拿大地址规则
- 加拿大地址解析器
- 加拿大 building / unit 双层清洗链路
- 加拿大外部参考源接入能力
- 加拿大历史数据训练样本

### 5.2 自部署能力

系统必须支持：

- 本地部署
- 私有服务器部署
- 离线运行
- 数据和模型完全由用户自己掌控

### 5.3 数据接入

系统必须支持用户接入自己的数据：

- 历史批量数据导入
- 增量数据持续拉取
- 用户自己的 reference 源
- 用户自己的标签和人工审核结果

### 5.4 训练与学习

系统必须支持：

- 模型训练
- 模型评估
- 影子运行
- gold set 冻结
- human gold 冻结
- 主动学习
- LLM 辅助标注

### 5.5 在线 API

系统必须支持对外 API：

- 地址解析
- 地址标准化
- 地址校验
- 单元补全建议
- building / unit 置信度输出
- review 建议输出

### 5.6 控制台

系统必须有一个控制台，负责：

- 启动 / 暂停增量接入
- 启动 / 暂停持续模式
- 触发一次增量清洗
- 触发训练 / 评估 / 冻结
- 查看任务状态和最近结果

控制台只负责控制和观测，不负责长时间执行。

## 6. 架构要求

系统应该拆成四个长期运行子系统：

1. **接入层**
   - 负责拉取新数据
   - 维护 cursor
   - 做幂等去重

2. **清洗层**
   - 负责 normalize / parse / publish / validate / user facts
   - 自动推进

3. **学习层**
   - 负责训练、评估、影子运行、LLM 回流、gold 冻结

4. **控制台**
   - 负责触发、暂停、恢复、查看状态

这四层要分离，但彼此联动。

## 7. 数据要求

系统必须保留以下数据层：

- 原始事实层
- 标准化层
- 解析候选层
- building / unit canonical 层
- 校验结果层
- 用户事实层
- 外部参考层
- gold label 层
- 模型运行层
- job / 控制台层

## 8. 质量要求

系统不能只看“跑了多少条”，还要看：

- building precision / recall
- unit precision / recall
- validation accuracy
- auto-enrich accuracy
- review rate
- GPS conflict rate
- gold set 覆盖率

## 9. 设计原则

1. 默认加拿大 / 北美优先
2. 开源自部署优先
3. 数据隔离优先
4. 模型版本化优先
5. 控制台可操作优先
6. 在线 API 和离线生产分离
7. ML / LLM 是主决策增强层，不是唯一真相源

## 10. 成功标准

系统成功的标准不是“全球都能用”，而是：

- 加拿大 / 北美地址处理准确
- 默认模型可直接使用
- 别人可以拿去改造自己的国家 / 区域
- 可以清洗自己的地址库
- 可以训练自己的模型
- 可以提供自己的地址解析 API
- 控制台、清洗、训练、评估都完整可运行

## 11. 说明

这是一套开源参考系统。  
它的价值在于：

- 自己可用
- 别人可改
- 架构完整
- 组件清晰
- 能覆盖“解析 / 清洗 / 生产 / API / 学习 / 控制台”全链路

## 12. Ingestion Service

Ingestion Service 的目标是把第三方私有地址数据接入系统，但**不把这些私有数据提交到 git**。

系统先支持两种接入方式：

1. **API Pull**
   - 第三方通过自己的私有 API 暴露新增原始地址
   - AddressForge 从 API 拉取数据
   - 第三方数据仍然保留在对方系统内

2. **Database Direct Import**
   - 第三方把原始地址导入到他们自己的数据库表
   - AddressForge 通过配置连接该数据库表
   - 读取新增行并进入后续清洗流水线

约束：

- 私有源文件、私有数据库配置、私有地址样本都不进入仓库
- 默认仓库只保存 schema、示例配置、文档和可执行代码
- 第三方可以自己决定是“提供 API”还是“提供数据库表”
