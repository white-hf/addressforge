# AddressForge 运营系统体验问题说明

## 问题背景
当前 AddressForge 运营系统的页面结构和入口，主要按照技术模块或系统能力划分，例如：

- `Batch Management`
- `Review`
- `Dashboard`
- `Reports`
- 各类 `job / train / evaluation / release gate`

这种组织方式对开发者和系统维护者是清楚的，因为它反映的是系统内部模块边界；但对运营人员来说，实际工作目标并不是“进入某个模块使用某个功能”，而是**完成一条业务流程**，例如：

- 生成待审核任务
- 完成人工审核
- 冻结 gold
- 触发训练
- 触发评测
- 查看 shadow / gate 结果
- 判断是否进入下一轮

运营人员关心的是“下一步该做什么”，而不是“这个功能在哪个模块里”。

## 当前体验上的核心问题

### 1. 用户需要自己理解系统流程
系统没有把完整业务路径直接呈现为连续步骤，用户必须自己知道：

- 审核完成后去哪里冻结 gold
- 冻结完后去哪里训练
- 训练后去哪里评测
- 评测后 shadow 是否自动发生
- gate 结果去哪里看

这要求用户先在脑中建立一套系统运行逻辑，再去不同页面之间跳转执行。

### 2. UI 更像功能后台，而不是业务工作台
当前界面更像：

- 一个按功能拆分的控制面板
- 一个面向系统模块的管理后台

而不是一个面向运营执行的工作台。结果是用户面对的是：

- 功能入口
- 报表入口
- job 入口
- 模块状态入口

而不是：

- “现在该做第几步”
- “当前卡在哪一步”
- “下一步应该点哪里”

### 3. 执行动作分散在多个页面
同一条业务链路上的动作，分散在不同区域：

- 审核在 `Review`
- gold freeze 在 `Batch`
- training / evaluation 在 `Dashboard`
- gate 结果在 `Reports`

这会导致用户在完成一个业务目标时，需要跨多个页面跳转，并在每次切换时重新判断上下文。

### 4. 用户需要自己记忆依赖关系
当前系统虽然技术上支持：

- freeze gold
- retrain
- re-evaluate
- shadow
- gate check

但这些关系没有以业务语言直接组织出来。用户需要自己知道哪些步骤是前置、哪些会自动 follow-up、哪些只是查看结果。

这意味着：

- 用户不是在“执行流程”
- 而是在“操作系统模块”

### 5. 对非技术运营人员心智负担偏高
如果用户不是系统开发者，而只是审核员、运营同学或项目执行人员，当前界面会带来明显心智负担：

- 需要理解系统术语
- 需要理解模块边界
- 需要理解 job 和流程之间的关系
- 需要自己判断某一步是否已经完成

这会降低执行效率，也提高误操作或漏步骤的风险。

## 本质问题
当前产品的信息架构主要回答的是：

- “系统有哪些功能模块？”

但运营人员真正需要的是：

- “我现在要完成的业务流程是什么？”
- “我现在处于哪一步？”
- “下一步应该做什么？”
- “哪些步骤已经自动完成，哪些需要我手动执行？”

因此，当前体验的本质问题不是单个按钮缺失，而是：

**产品是按系统能力组织的，不是按运营执行流程组织的。**

## 直接影响
这会直接带来以下体验问题：

- 用户需要思考，而不是顺着流程执行
- 用户需要跨模块跳转完成一项工作
- 用户需要自己记忆流程依赖关系
- 用户不容易快速判断“当前阶段是否完成”
- 系统虽然功能齐全，但执行体验不够顺手

## 总结
当前运营系统更像“模块化功能后台”，而不是“面向业务执行的流程工作台”。用户要完成一条业务链路时，必须自己理解流程、自己跨页面找功能、自己判断下一步，而不是被系统自然地一步步带着完成。
